# jeff:init — Edge Cases

## Reinitializing a working `.jeff/`

If the user has been using jeff for weeks, their `STORY_MAP.md` / `OPPORTUNITIES.md` / `HYPOTHESES.md` contain real work, not template placeholders. **Never overwrite a non-template file.**

Detection: read the file; if it contains HTML comments like `<!-- TODO -->` and no user-written prose, it's still a template. Otherwise, it has real content — preserve it.

If the user wants a fresh `.jeff/`, make them delete it explicitly first. Don't be the skill that loses their discovery notes.

## Running inside a monorepo

If several packages live under one repo, `.jeff/` goes at the package level, not the repo root, unless the user says otherwise. Ask: "Should discovery live for the whole repo, or just the package you're in?"

## Directory name looks like a scratch pad

Names like `tmp`, `test`, `playground`, or autogenerated strings (`tmp.abc123`) probably don't want to become the project_name in a config file. Ask for an override.

## Git-ignored or read-only working directory

`.jeff/` is plaintext markdown — it should normally be committed. If `.gitignore` excludes `.jeff/`, warn the user; they likely don't want that. If the working directory is read-only (e.g. inside a release tarball), stop.

## Multiple users initializing concurrently

Not a real concern for local use, but in shared CI, two runs scaffolding at once can corrupt a file mid-write. If `.jeff/config.yaml` exists and was created within the last 10 seconds, pause and re-check — another process may be scaffolding.

## Template drift between skill and docs

The `templates/` folder is the single source of truth. If the README or another skill documents a different template, update the README — don't edit the templates to match stale docs.
